# Задача: экран «Пользователи» (список + редактирование)

Реализовать внутри приложения фронтенд-экран управления пользователями. Задача умышленно небольшая по объёму, но должна быть выполнена «по-продакшену»: с валидациями, форматированием, состояниями загрузки/ошибок, реактивной логикой полей и аккуратным UX. Технология реализации не задаётся — используйте средства того стека, который тестируется.

## Контракт данных и API

Модель пользователя, справочник ролей и весь серверный API (эндпоинты, пагинация, фильтры, сортировка, методы записи, заголовок с общим количеством) описаны в [README.md](./README.md). **Берите поля и эндпоинты оттуда, не переопределяйте контракт в коде.**

Опорные факты, на которые ссылается задача (детали — в README.md):
- Ресурс `users` и справочник `roles`.
- Постраничная выдача через query-параметры пагинации; общее число записей — в заголовке ответа.
- Фильтрация по полям (используется для проверки уникальности) и методы записи (создание/замена/частичное обновление/удаление).
- Запись «фейковая»: сервер отвечает корректными статусами, но изменения не персистятся — UI должен вести себя так, как будто они применились.

## Часть 1. Список пользователей

- Постраничный список с **серверной пагинацией** (параметры и общее количество — по README.md). Переключение страниц не подгружает лишнего.
- В строке отображаются основные поля пользователя и его состояние доступа, плюс действия над строкой.
- Действия в строке (inline, без ухода со списка):
  - **Заблокировать / Разблокировать** — переключает состояние доступа пользователя. Действие асинхронное; применяется оптимистично, при ошибке — откат к прежнему значению и сообщение. Блокировка запрашивает подтверждение.
  - **Редактировать** — открывает окно редактирования пользователя.
- Заблокированный пользователь визуально отличается в списке.
- Явные состояния списка: **загрузка**, **ошибка** (с возможностью повторить запрос), **пустой список**.
- Пока выполняется действие над строкой, повторный запуск того же действия по этой строке недоступен.

## Часть 2. Окно редактирования

Окно редактирует данные одного пользователя. Набор редактируемых полей — из контракта в README.md; поля, помеченные как read-only (напр. идентификатор и дата создания), не редактируются.

### Правила валидации

- **Имя** — обязательное. Только буквы, пробел и дефис; без цифр. Длина 2–50.
- **Логин (username)** — обязательное. Нижний регистр, латиница/цифры/подчёркивание, без пробелов; форматируется по мере ввода. Дополнительно **асинхронная проверка уникальности** через фильтрацию по этому полю на сервере.
- **Email** — обязательное. Проверяется регулярным выражением на формат. Дополнительно **асинхронная проверка уникальности** через фильтрацию по email на сервере; на время проверки поле показывает индикатор, конфликт — отдельная ошибка «email уже используется».
- **Телефон** — обязательное. Вводится с **маской/форматированием** по мере ввода (единый формат, напр. `+7 (999) 999-99-99`). Валидируется на полноту и корректность. Наружу (в данные) уходит в нормализованном виде, отдельно от отображаемого.
- **Роль** — обязательное, выбор из значений справочника ролей (список подгружается из соответствующего ресурса, а не хардкодится).
- **Аватар** — необязательное; если заполнено, значение должно быть корректным URL.

### Поведение окна

- Валидация по потере фокуса и при попытке отправки; ошибки показываются под соответствующими полями понятным текстом.
- Кнопка сохранения недоступна, пока форма невалидна или пока идёт отправка.
- Отслеживание изменений (dirty): при попытке закрыть окно с несохранёнными изменениями — предупреждение.
- Отправка асинхронная: на время сохранения форма заблокирована и показывает прогресс; **ошибка сервера показывается пользователю** и не закрывает окно; успех — окно закрывается, строка в списке обновляется.
- Закрытие окна: кнопкой, по `Esc`, по клику вне окна (с учётом предупреждения о несохранённых изменениях).
- При открытии фокус переходит на первое поле; фокус удерживается внутри окна; при закрытии возвращается на элемент, из которого окно открыли.

## Общие требования (то, что отличает продакшен)

- Чёткое разделение слоёв: данные / логика (валидация, зависимости полей, форматирование) / представление. Логика не «зашита» в разметку.
- Валидаторы и форматтеры оформлены как переиспользуемые единицы, а не дублируются по месту.
- Доступность: подписи к полям, связь ошибок с полями, работа с клавиатуры, корректный фокус в окне.
- Отсутствие «скачков» верстки при появлении состояний загрузки/ошибок.
- Обработка ошибок присутствует на каждом асинхронном участке (список, действия строки, проверки уникальности, сохранение).
- Все пользовательские строки вынесены так, чтобы их можно было локализовать (без хардкода в разметке).
- Модель данных типизирована; код пригоден к тестированию.

## Вне рамок задачи

- Реальный бэкенд, авторизация, маршрутизация между экранами.
- Создание и удаление пользователей (эндпоинты существуют, но в этой задаче не требуются).
- Богатое визуальное оформление — достаточно опрятного минимума.
- Конкретная библиотека/фреймворк не навязываются: решение выполняется средствами тестируемой технологии.

## Критерии приёмки (короткий чек-лист)

1. Список грузится порциями (серверная пагинация по README.md), корректно листается, показывает загрузку/ошибку/пусто.
2. Блокировка/разблокировка работает оптимистично, откатывается при ошибке, блокировка подтверждается.
3. Окно редактирования валидирует все поля по описанным правилам, включая async-уникальность email и логина.
4. Логин форматируется по мере ввода; телефон вводится по маске и нормализуется в данных.
5. Роль выбирается из справочника, подгруженного с сервера.
6. Сохранение недоступно при невалидной форме; ошибки сервера показываются; успех обновляет список.
7. Несохранённые изменения защищены предупреждением; фокус и клавиатура в окне работают корректно.
